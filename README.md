# Звіт з лабораторної роботи №2
**Тема:** Локальне сховище даних та стратегія взаємодії з API

---

## Мета роботи
Надати застосунку "Water and Vitamin Tracker" можливість працювати в offline-режимі шляхом впровадження локальної бази даних. Розробити комунікаційну стратегію з сервером, визначити API-контракт та реалізувати архітектурний патерн Repository для реактивного оновлення інтерфейсу.

---
## 1. Локальне сховище (Room)

В якості сховища обрано структуровану базу даних на базі **Room (SQLite)**. Це рішення дозволяє ефективно працювати з двома незалежними таблицями (`vitamins` та `water_records`) та гарантує типобезпеку під час компіляції.

У DAO-інтерфейсах (`VitaminDao`, `WaterDao`) реалізовано всі 4 необхідні базові операції (CRUD):
* **Збереження:** `insertVitamin()` / `insertWaterRecord()` із політикою перезапису `OnConflictStrategy.REPLACE`.
* **Читання списку:** `getAllVitamins()` / `getAllWaterRecords()`, які повертають реактивні потоки `Flow`.
* **Читання одного об'єкта:** `getVitaminById()` / `getWaterRecordById()`.
* **Видалення:** `deleteVitamin()` / `deleteWaterRecord()`.

До моделей даних було додано поле `syncStatus: String` зі значенням за замовчуванням `"pending"` для відстеження стану синхронізації з бекендом.

---

## 2. API-контракт та Стратегія комунікації

### Вибрана стратегія: Offline-first (Оптимістичне збереження)
Застосунок спроєктований так, щоб користувач міг відмічати випиту воду або прийняті вітаміни навіть без доступу до Інтернету. 
1. При додаванні нового запису він **миттєво** зберігається в локальну БД.
2. Паралельно ініціюється виклик до API.
3. Залежно від успішності мережевого запиту, статус запису в локальній БД оновлюється на `synced` (успіх) або `error` (помилка).

### API-контракт (REST)
Для імітації серверної частини створено клас `MockApiService` із штучною затримкою `delay(1000)`.

| Метод | Ендпоінт (Умовно) | Опис операції | Тіло запиту / Параметри | Відповідь |
| :--- | :--- | :--- | :--- | :--- |
| **POST** | `/api/vitamins` | Додавання нового вітаміну на сервер. | JSON об'єкт `Vitamin` | `200 OK`, `Boolean` |
| **DELETE**| `/api/vitamins/{id}` | Видалення вітаміну з бази сервера. | Шлях: `id` (Int) | `200 OK`, `Boolean` |
| **POST** | `/api/water` | Збереження запису про випиту воду. | JSON об'єкт `WaterRecord` | `200 OK`, `Boolean` |
| **DELETE**| `/api/water/{id}` | Видалення запису про воду. | Шлях: `id` (Int) | `200 OK`, `Boolean` |

## 3. Архітектура (Repository та Реактивний UI)

Логіка поєднання локального сховища та мережевого шару інкапсульована у класі `AppRepository`. Він слугує єдиним джерелом правди (Single Source of Truth) для `AppViewModel`.

**Як працює зв'язок з інтерфейсом:**
1. Дані з бази Room транслюються у вигляді `Flow<List<T>>`.
2. У `AppViewModel` ці потоки конвертуються у `StateFlow` за допомогою оператора `stateIn`.
3. UI-компоненти (наприклад, `VitaminListScreen`) підписані на ці стани через `collectAsState()`.
4. Коли користувач натискає кнопку додавання вітаміну (FAB `+`), запис миттєво з'являється у списку завдяки реактивності бази даних, **без перезавантаження екрана** чи мануального оновлення масивів. Якщо закрити застосунок (вбити процес) і відкрити знову, дані залишаються на екрані.
